  ---
  二、N字结构的核心优势

  1. 形态逻辑有交易直觉支撑

  N字结构（A低→D高→B更高低→C突破进场）本质上是趋势中的回调买入模式。A-B
  形成上升趋势线，D
  是关键阻力位。这个逻辑比海龟的纯突破更精细——它在趋势确认（更高低点 B >
  A）后才进场。

  2. 实现轻量、迭代快

  纯 pandas 实现，不依赖
  Backtrader，回测速度快。参数扫描框架（scan_n_structure.py）支持 IS/OOS
  分离验证，实验迭代效率高。

  3. 调优后的表现数字亮眼

  CAGR 18.5%、MDD 8.6%、957 笔交易、全部 6
  品种盈利——这些数字如果真实可靠，已经超越海龟策略。

  ---
  三、关键缺陷与风险（按严重程度排列）

  🔴 P0 — N字结构识别存在"未来信息"问题

  这是最根本的问题。find_n_structure_in_window 的逻辑：

  python
  # A = 窗口内最低价  → 需要看过整个窗口才知道哪里是最低
  a_local_idx = window['low'].idxmin()

  # D = A 之后最高价  → 需要看过 A 之后的所有数据
  d_local_idx = after_a['high'].idxmax()

  # B = D 之后最低价  → 同理
  b_local_idx = after_d['low'].idxmin()

  问题：在实时交易中，当 bar i 到来时，你只能看到 0~i 的数据。但代码在第 i
  天扫描 [i-100, i) 窗口，这个窗口内的 A/D/B 三个点都依赖窗口内全局
  min/max——也就是说，只有窗口"走完"后才知道 A 在哪里。

  举例：假设窗口前半段有个低点被认为是
  A，但窗口最后几天出现了更低的低点——那真正的 A 就变了，之前基于"假
  A"的所有判断都作废。这种"事后诸葛亮"式的形态识别，会系统性地高估策略表现。

  这不是"防偷价"（代码里 B < end_idx-1 只是确保 B
  不是最后一天），而是形态识别算法层面的信息泄露。

  🔴 P0 — 没有日频净值曲线

  compute_metrics 的 Sharpe/MDD 计算只基于交易序列的 PnL，不是基于每日净值：

  python
  equity = initial_capital + np.cumsum(pnls)  # 按交易序列累积

  这导致两个问题：
  - MDD 严重低估：只计算了每笔交易结束后回撤，忽略了持仓期间的盘中回撤
  - Sharpe 计算不可比：用 252 / avg_holding_days 做年化调整是非常粗糙的近似

  海龟策略用 Backtrader 的 TimeReturn analyzer 算的是真正的日频
  Sharpe。这两个数字不能直接对比。

  🟡 P1 — 无摩擦成本

  N字策略的成交价是精确的开盘价，没有滑点、没有手续费。海龟策略有
  commission_pct: 0.00015 和 slippage_pct: 0.001。

  957 笔交易 × 0.1% 滑点 ≈ 约 1 个全仓周转的摩擦成本。加上每笔 0.015%
  手续费，实际 CAGR 可能要扣 1-2 个百分点。

  🟡 P1 — 每品种独立回测，无组合层面

  run_on_multi 只是对每个品种分别跑 run()，然后用固定 capital_per_symbol =
  100000/6。这忽略了：
  - 多品种同时持仓时的资金竞争
  - 组合层面的风险预算
  - 品种间的相关性对冲

  海龟策略在 TurtleStrategy
  中有完整的组合管理：集中度熔断、全账户风险敞口上限、T+1 约束等。

  🟡 P2 — S23 文档已列出的风控缺陷

  你自己在 S23 文档中已经诚实地列出了（这是好习惯），但它们是真实的风险：

  1. 滑点/跳空模型缺失
  2. 固定本金而非动态权益计算仓位
  3. 无总敞口上限
  4. 加仓不重算单位风险
  5. 无连续亏损熔断
  6. MDD 偏乐观（你估计实盘 12-15%，代码显示 8.6%）

  🟡 P2 — 退出逻辑过于复杂

  5 种退出原因 + 再进场机制：
  - 初始止损
  - 跟踪止损（D 突破后）
  - B 点结构失效（收盘跌破 B）
  - D 点突破失败（5 K 线超时）
  - D 点突破成功后的跟踪止损

  问题不在逻辑本身，而在于参数多、自由度大：
  - 5 K 线超时为什么不是 3 或 7？
  - B 点跌破 0.95 为什么不是 0.97？
  - 跟踪止损 5.0×ATR 为什么不是 4.0？

  957 笔交易给了参数足够的"拟合空间"。OOS 验证（2014-2019 跑出 CAGR 22.7% > IS
  19.3%）是个好信号，但单一 OOS 区间不够。

  1. 两套策略的关系不清晰

  架构文档完全没有提到 N字结构。README 里的项目结构也没列

  - 互补关系（组合）？
  - N字最终要替代海龟？

  我的建议：不要把它们看作两个独立的策略，而是考虑融合：
  - N字结构作为海龟的入场信号增强器——当海龟 20 日突破同时出现
  N字结构确认时，加大仓位
  - 或者作为独立子系统，与海龟按 60/40 或其他比例分配资金

  2. 缺少实盘对接

  海龟策略有完整的 daily_signal.py + state.json

  五、具体的改进建议

  优先级 1（立刻做）：修复 N字结构识别

  将形态识别从"窗口全局极值"改为"增量确认"模式：

  方案 A：分段识别
    - 在窗口中找 A（局部低点，不要求全局最低）
    - A 之后找 D（局部高点，高于 A 一定比例）
    - D 之后找 B（局部低点，高于 A 且低于 D）
    - 要求 A/D/B 之间有一定间距

  方案 B：引入确认延迟
    - A 确认：低点后 N 根 K 线未创新低
    - D 确认：高点后 N 根 K 线未创新高
    - B 确认：低点后 N 根 K 线未创新低 + 高于 A

  这样每个点都是"在当时可以确认的"，消除未来信息泄露。

  优先级 2（短期）：完善基础设施

  1. 日频净值曲线：在 run() 中返回每日的 mark-to-market 净值，用于正确的
  Sharpe/MDD 计算
  2. 摩擦成本：加入 slippage_pct 参数，进场/出场价格加上滑点
  3. 组合回测：构建一个简单的组合回测，共享资金池，含总敞口限制
  4. 编写测试：至少覆盖
  find_n_structure_in_window、compute_metrics、NStructureStrategy.run

  优先级 3（中期）：架构整合

  1. 在架构文档中加入 N字策略
  2. 实盘信号对接：写 scripts/daily_signal_n_structure.py
  3. 策略融合实验：开一个实验分支，探索海龟+N字的组合方式

  优先级 4（长期）：策略深化

  1. 多时间框架 N字：日线找结构，小时线找精确入场
  2. 市场状态适配：震荡市用 N字（回调买入），趋势市用海龟（突破追入）
  3. 品种特化参数：中证500 的 N字参数 vs 黄金ETF 的 N字参数分别优化
  4. 更多 OOS 验证：滚动窗口 OOS、不同市场环境分段验证
