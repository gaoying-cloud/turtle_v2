# Claude Code 学习路线图

> 适合：有编码经验但 Claude Code 新手
> 目标：从"不知怎么用"到"成为日常开发的高效伙伴"

---

## 路线图总览

```
Level 1: 基础操作  ─── 能问问题、看懂回答、控制会话
       ↓
Level 2: 日常开发  ─── 能独立写完一个功能，用 AI 辅助调试
       ↓
Level 3: 系统掌控  ─── 能指挥多 Agent、自动化流程、定制配置
       ↓
Level 4: 高阶融合  ─── 能把 Claude Code 嵌入团队工作流
```

**学习方法**：每个 Level 按 **学习 → 练习 → 评估 → 通关** 循环推进，不跳到下一级。

---

## Level 1：基础操作 — "会用"

> 目标：能启动会话、提出有效问题、理解回答、管理上下文

### 学习内容

| 模块 | 具体知识点 |
|------|-----------|
| **会话启动** | 如何打开 Claude Code（CLI / VS Code / JetBrains） |
| **提问技巧** | 给上下文：贴代码、贴报错、说清楚"我想要什么 + 我试过什么" |
| **理解回答** | Claude 的回答结构：解释 → 代码 → 建议 → 反问澄清 |
| **`!` 前缀** | `! py test.py` — 在会话中执行本地命令，输出直达对话 |
| **基础命令** | `/help`、`/clear`、`/model`、`/fast` |
| **权限系统** | 工具调用会弹权限确认，理解"允许/拒绝/记住"的区别 |

### 练习任务

- [ ] 启动 Claude Code，输入一句话问题（如"这个项目结构是怎样的"）
- [ ] 用 `!` 执行一条 git 命令查看状态
- [ ] 故意问一个模糊问题，观察 Claude 如何追问澄清
- [ ] 跑 `/help` 看有哪些命令
- [ ] 跑 `/model` 看当前模型
- [ ] 跑 `/clear` 清空一次上下文

### 评估标准

| 检查项 | 通过标准 |
|--------|---------|
| 能启动 Claude Code | 3 种方式中至少 1 种（CLI / VS Code / JetBrains） |
| 能提出有效问题 | 包含：代码片段 / 报错信息 / 期望结果 / 已尝试方案，至少 3 项 |
| 能执行本地命令 | 用 `!` 成功执行至少 1 条 shell 命令并看到输出 |
| 能使用基础命令 | `/help` `/clear` `/model` 都能跑通 |

**✅ 通关判断**：以上 4 项全部通过 → 进入 Level 2

## Level 2：日常开发 — "好用"

> 目标：能用 Claude Code 辅助编码、调试、测试、提交，形成闭环

### 学习内容

| 模块 | 具体知识点 |
|------|-----------|
| **代码生成** | 用自然语言描述功能需求，让 Claude 生成代码 |
| **代码审查** | `/code-review` — 让 Claude 审查你的 diff，找 bug 和改进点 |
| **问题修复** | 贴报错 → 描述现象 → Claude 定位根因 → `/fix` 应用修复 |
| **测试辅助** | `! py -m pytest tests/ -q` 跑测试，把失败结果贴回给 Claude |
| **代码简化** | `/simplify` — 自动优化代码质量（复用、简化、效率） |
| **任务追踪** | `/tasks` — 创建 todo list，管理多步骤任务进度 |
| **计划模式** | `/plan` — 复杂功能先设计再实施，避免白干 |
| **`/verify`** | 端到端验证改动是否真的按预期工作 |

### 练习任务

用一个**你熟悉的项目**（或这个海龟策略项目）完成一轮完整开发循环：

- [ ] **需求**：描述一个小功能（如"加一个日志配置函数"），让 Claude 生成代码
- [ ] **审查**：生成后跑 `/code-review medium` 检查质量
- [ ] **修复**：如果有问题，描述现象让 Claude 修，或跑 `/fix`
- [ ] **测试**：`! py -m pytest tests/ -q` 跑测试，失败结果贴给 Claude
- [ ] **简化**：跑 `/simplify` 优化代码
- [ ] **验证**：跑 `/verify` 确认改动正常
- [ ] **追踪**：用 `/tasks` 列一下今日待办
- [ ] **计划**：对一个不那么确定的功能先用 `/plan`

### 评估标准

| 检查项 | 通过标准 |
|--------|---------|
| 能独立完成"需求→编码→审查→测试→提交"闭环 | 完整跑通一次，全部 8 个练习任务完成 |
| 能看懂 Claude 生成的代码并做修改 | 至少手动调整过 1 处生成代码 |
| 能利用测试结果驱动修复 | 跑测试 → 贴失败 → 修复 → 再跑通过 |
| 会使用任务列表管理多步骤工作 | `/tasks` 创建过至少 3 个任务并标记完成 |

**✅ 通关判断**：以上 4 项全部通过 → 进入 Level 3

## Level 3：系统掌控 — "高效"

> 目标：能定制 Claude Code 的行为、用多 Agent 并行工作、建立自动化流程

### 学习内容

| 模块 | 具体知识点 |
|------|-----------|
| **配置定制** | `/config` 修改设置，`/fewer-permission-prompts` 减少弹窗 |
| **内存系统** | `/remember` 保存跨会话记忆，理解 `memory/` 目录结构 |
| **项目文档** | `/init` 初始化 `CLAUDE.md`，编写项目级指引（关键！） |
| **多 Agent 并行** | 用 Agent 工具启动后台任务，同时处理多个独立工作 |
| **工作流编排** | 用 Workflow 工具编排多 Agent 流水线（review + verify + fix 串联） |
| **循环任务** | `/loop` 设定周期性任务，Cron 定时任务持久化 |
| **工作树隔离** | Worktree 隔离并行开发，互不干扰 |
| **`!` 高级用法** | 管道、重定向、交互式命令 |

### 练习任务

- [ ] **配置**：跑 `/fewer-permission-prompts` 减少弹窗，确认生效
- [ ] **记忆**：用 `/remember` 保存一条项目关键决策，新会话中验证能 recall
- [ ] **CLAUDE.md**：用 `/init` 生成或手动优化项目的 `CLAUDE.md`
- [ ] **多 Agent**：同时发两个独立任务给不同的 Agent（如一个审查代码，一个搜索资料）
- [ ] **自动化**：用 `/loop 30m /code-review low` 设一个半小时的循环审查
- [ ] **隔离开发**：用一个 Worktree 做实验性修改，不污染主分支

### 评估标准

| 检查项 | 通过标准 |
|--------|---------|
| 能自定义配置减少干扰 | 权限弹窗减少 50% 以上 |
| 能有效使用记忆系统 | 跨会话 recall 成功，记忆文件结构理解正确 |
| 能编写有效的 CLAUDE.md | 包含项目概览、命令速查、架构层级、关键设计决策 |
| 能同时推进多个任务 | 成功启动并完成至少 2 个并行 Agent 任务 |
| 能设定自动化任务 | 成功运行一次 `/loop` 或 Cron 定时任务 |

**✅ 通关判断**：以上 5 项全部通过 → 进入 Level 4

---

## Level 4：高阶融合 — "精通"

> 目标：把 Claude Code 嵌入团队工作流，建立体系化的 AI 辅助开发流程

### 学习内容

| 模块 | 具体知识点 |
|------|-----------|
| **CI/CD 集成** | 在 CI 中调用 Claude Code 做自动审查、自动生成报告 |
| **自定义 Hook** | 配置 pre-commit hook 自动审查 staged 变更 |
| **团队规范** | 制定团队级 `CLAUDE.md` 模板，统一 AI 交互规范 |
| **复杂工作流** | 用 Workflow 编排"发现 → 审查 → 修复 → 验证"全自动管道 |
| **Code Review 自动化** | PR 自动触发 Claude Code review，输出结构化审查报告 |
| **性能调优** | 理解 Token 消耗、上下文窗口管理、模型选择策略 |
| **故障排查** | 当 Claude 给出错误答案时，能诊断原因（上下文不足 / 权限不够 / 模型限制） |
| **Agent 角色设计** | 针对不同任务类型（安全审查员 / 代码简化师 / 测试工程师）设计不同的 Agent 提示词 |

### 练习任务

- [ ] **CI 集成**：在 GitHub Actions 中配置 PR 自动触发 `/code-review`（通过 `gh` CLI 或 API）
- [ ] **Hook**：配置一个 pre-commit hook 在提交前自动跑 `/security-review`
- [ ] **团队模板**：编写一份团队通用的 `CLAUDE.md` 模板
- [ ] **自动化管道**：设计并跑通一个 Workflow：扫描代码 → 并行多维度审查 → 汇总报告 → 自动修复简单问题
- [ ] **诊断复盘**：回顾一次 Claude 给错答案的案例，分析根因并记录下来避免重犯

### 评估标准

| 检查项 | 通过标准 |
|--------|---------|
| 能在 CI 中调用 Claude Code | 至少在一个 CI pipeline 中集成成功 |
| 能设计团队规范 | 输出一份可复用的 CLAUDE.md 模板 |
| 能编排复杂工作流 | 跑通至少 3 阶段以上的自动化管道 |
| 能诊断 Claude 的错误 | 能定位至少 2 种常见失败模式及其解决方案 |
| 能管理 Token/成本 | 理解 Token 消耗模型，能估算单次操作的成本 |

**✅ 通关判断**：以上 5 项全部通过 → 🎉 你已经是一名 Claude Code 高阶用户了

## 附录

### A. 常见失败模式 & 解决方法

| 现象 | 根因 | 解决 |
|------|------|------|
| Claude 回答不准确 | 上下文不够 | 贴更多相关代码，用 `/clear` 后重新提问 |
| Claude 遗忘早期约定 | 上下文窗口满了 | 用 `/remember` 保存关键决策，或重启会话 |
| 权限弹窗太多 | 未配置白名单 | 跑 `/fewer-permission-prompts` |
| 生成代码有未来函数 | 未提供足够上下文 | 在 prompt 中强调"只能用已有数据，不能 look ahead" |
| 回复太慢 | 模型太重 | 跑 `/fast` 或切换轻量模型 |
| Claude 不理解项目结构 | 无 CLAUDE.md | 跑 `/init` 初始化项目文档 |

### B. 推荐学习顺序（按投入时间）

```
🕐 30 分钟 → Level 1 基础操作
🕐 2 小时   → Level 2 日常开发（完成一次完整开发闭环）
🕐 4 小时   → Level 3 系统掌控（配置 + 多 Agent + 自动化）
🕐 持续     → Level 4 高阶融合（团队流程 + CI + 调优）
```

### C. 一句话原则

> **把 Claude Code 当作一个会编程的队友，而不是搜索引擎。**
> 给它上下文，告诉它你要什么，让它生成方案，你来判断和修改。
